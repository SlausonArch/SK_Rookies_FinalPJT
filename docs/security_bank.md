# 보안 취약점 진단 및 조치 보고서 — 은행 (Bank)

- **대상 URL**: http://localhost:25173
- **작성일**: 2026-03-27
- **대상 기술 스택**: React (Vite) + Spring Boot 백엔드 공유

---

## 개요

은행 사이트는 거래소와 동일한 프론트엔드 코드베이스(`frontend/`)와 백엔드(`Spring Boot`)를 공유하며, `VITE_APP_MODE=bank`로 빌드되어 은행 관련 기능(계좌 조회, 입출금, 이체)만 표시됩니다. 별도의 Docker 컨테이너(`bank-front`)로 격리되어 운영됩니다.

## 1. XSS / CSRF

### XSS (Cross-Site Scripting)

| 항목 | 내용 |
|------|------|
| 해당 여부 | **해당 사항 없음** |

은행 사이트는 커뮤니티 게시글 작성·상세 기능이 없으므로 사용자 입력 콘텐츠를 HTML로 렌더링하는 경로가 없음. 입출금·이체 관련 입력값(금액, 계좌번호)은 숫자·고정 형식이며 HTML 렌더링 없음.

### CSRF (Cross-Site Request Forgery)

| 항목 | 내용 |
|------|------|
| 해당 여부 | **해당 사항 없음 (구조적으로 방어됨)** |

거래소 사이트와 동일하게 JWT + SameSite=Strict 쿠키 구조로 방어.

## 2. Injection

| 항목 | 내용 |
|------|------|
| 해당 여부 | **해당 있음 → 조치 완료** |

**기존 방식**
JPA 파라미터 바인딩으로 SQL Injection 기본 방어 중.

**조치 내용**
- 입출금·이체 금액: 숫자 타입(`Long`/`BigDecimal`) 직접 바인딩, 문자열 파싱 없음.
- 계좌번호: `@Pattern(regexp = "^\\d+$")` 검증으로 숫자 이외 문자 차단.
- 모든 DB 조회는 JPA Named Parameter 사용, SQL 직접 조합 없음.

## 3. 파라미터 / 히든 값 조작

| 항목 | 내용 |
|------|------|
| 해당 여부 | **해당 있음 → 조치 완료** |

**기존 방식 (취약 가능성)**
이체 요청 시 `fromAccountId`, `toAccountId` 등 계좌 식별자를 요청 파라미터로 전달할 경우, 타인 계좌 ID 변조 가능성 존재.

**조치 내용**
- 입출금·이체 API에서 `@AuthenticationPrincipal`로 로그인 사용자 식별.
- 서버에서 해당 계좌가 로그인 사용자 소유인지 재확인 후 처리.
- 금액 필드: 양수 정수 범위 검증, 음수 입력 차단.
