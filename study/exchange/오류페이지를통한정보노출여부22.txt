================================================================================
[VCE-EX-22] 오류 페이지를 통한 정보 노출 여부
도메인: exchange
위험도: Medium
================================================================================

1. 진단 항목 개요
--------------------------------------------------------------------------------
의도적으로 오류를 유발하는 요청을 전송하여, 서버의 오류 응답 페이지 및 메시지에서
내부 시스템 정보(기술 스택, DB 종류, 파일 경로, 클래스명, SQL 쿼리 등)가
노출되는지 검증한다.

Spring Boot의 Whitelabel Error Page, 미처리된 예외 스택 트레이스, SQL 에러 메시지
등이 응답에 포함되면 공격자는 추가 공격(SQL Injection, 경로 순회 등)에 활용한다.

위험도가 Medium인 이유:
- 단독으로는 직접적인 침해로 이어지지 않으나, 정보 수집 단계에서 핵심 역할
- SQL 에러 메시지 노출 시 Blind SQLi → Error-Based SQLi로 전환 가능
- 스택 트레이스로 내부 로직 파악 후 타깃 공격 가능


2. 대상 기능 & 엔드포인트
--------------------------------------------------------------------------------
  - GET /api/nonexistent-path           (존재하지 않는 경로)
  - GET /api/members/invalid-type       (타입 불일치)
  - POST /api/auth/login                (잘못된 JSON 바디)
  - GET /api/orders?page=-1             (잘못된 파라미터 범위)
  - GET /api/members?search=<SQL>       (SQL 특수문자 주입)
  - GET /error                          (Spring Boot 기본 에러 페이지)
  - GET /api/404test                    (404 Not Found 응답 내용)
  - GET /api/members/999999999          (존재하지 않는 리소스)


3. 취약점 원리 (Why this works)
--------------------------------------------------------------------------------
[Spring Boot Whitelabel Error Page]
Spring Boot는 기본적으로 /error 경로에 HTML 형식의 에러 페이지를 렌더링.
페이지에 timestamp, status, error, exception, message, trace 등이 포함될 수 있음.
특히 spring.boot.admin 또는 개발 모드에서 전체 스택 트레이스 노출.

[ExceptionHandler 미구성]
@RestControllerAdvice(전역 예외 핸들러) 미구성 시, Spring이 기본 에러 응답을
반환하며 exception 클래스명과 메시지가 포함됨.

[application.properties 잘못된 설정]
  server.error.include-stacktrace=always
  server.error.include-message=always
  server.error.include-exception=true
위 설정 조합 시 스택 트레이스 전체가 JSON 응답에 포함됨.

[SQL 에러 메시지 직접 전달]
JPA/Hibernate 예외를 @ExceptionHandler에서 잡지 못해 DataAccessException이
그대로 클라이언트에 노출되는 경우, SQL 쿼리와 파라미터까지 포함될 수 있음.

5. 취약 여부 판단 기준
--------------------------------------------------------------------------------

취약 (Vulnerable):
  - 에러 응답에 Java 클래스명 또는 패키지 경로 포함 (com.vce.exchange.*)
  - 에러 응답에 "trace" 또는 "exception" 필드 포함
  - SQL 쿼리 또는 DB 에러 메시지 직접 노출
  - Spring Boot Whitelabel Error Page가 운영 환경에서 노출
  - 내부 파일 시스템 경로 노출 (/home/ubuntu/vce/backend/...)
  - DB 벤더 정보 노출 (MySQL, PostgreSQL 버전 등)

안전 (Safe):
  - 모든 에러 응답이 표준화된 형식으로 반환 (code + message만)
  - "trace", "exception", "path" 필드가 응답에 미포함
  - SQL 에러가 제네릭 메시지로 대체되어 반환
  - Whitelabel Error Page 비활성화 (404 시 커스텀 페이지 반환)
  - 개발/운영 환경 모두 동일한 에러 응답 형식 적용


6. 예상 취약 포인트 (코드 위치)
--------------------------------------------------------------------------------

[application.properties]

7. 권고 조치
--------------------------------------------------------------------------------

[즉시 조치]
1. application.properties 설정 변경:
   server.error.include-stacktrace=never
   server.error.include-message=never
   server.error.include-exception=false
   server.error.whitelabel.enabled=false

2. 전역 예외 핸들러 구현:
   @RestControllerAdvice
   public class GlobalExceptionHandler {
       private static final Logger log = LoggerFactory.getLogger(GlobalExceptionHandler.class);

       @ExceptionHandler(Exception.class)
       public ResponseEntity<ErrorResponse> handleAll(Exception e) {
           log.error("Unhandled exception: ", e);  // 서버 로그에만 기록
           return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
               .body(new ErrorResponse("SERVER_ERROR", "서버 오류가 발생했습니다."));
       }

       @ExceptionHandler(EntityNotFoundException.class)
       public ResponseEntity<ErrorResponse> handleNotFound(EntityNotFoundException e) {
           return ResponseEntity.status(HttpStatus.NOT_FOUND)
               .body(new ErrorResponse("NOT_FOUND", "리소스를 찾을 수 없습니다."));
       }
   }
