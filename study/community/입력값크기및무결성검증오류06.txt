1. 진단 항목 개요
--------------------------------------------------------------------------------
입력값 검증 오류는 서버가 사용자로부터 받는 데이터의 크기, 형식, 허용 문자 등을
충분히 검증하지 않을 때 발생하는 취약점이다.

주요 위험 유형:
  1. 크기 제한 미적용 → 과도한 데이터 전송으로 서버 자원 고갈(DoS)
  2. NULL 바이트 삽입 → 문자열 처리 로직 혼란 (경로 절단 등)
  3. 특수문자/이모지 미검증 → DB 인코딩 오류 또는 렌더링 문제
  4. 정수 오버플로우 → 음수 값 또는 예상 범위 외 값으로 로직 우회
  5. 빈 문자열/공백만 입력 → 제목 없는 게시글 생성 가능

VCE 커뮤니티에서는 게시글 제목/내용의 최대 길이 제한 및 특수 입력에 대한
서버 사이드 검증이 불충분할 것으로 추정된다. 프론트엔드 검증은 쉽게 우회되므로
반드시 백엔드 검증이 필요하다.

영향 범위:
  - 서버 메모리/CPU 과부하 (DoS 가능)
  - DB 컬럼 길이 초과로 데이터 절단 또는 에러
  - 특수 입력으로 인한 예외 미처리 → 정보 노출
  - NULL 바이트로 인한 파일 경로 절단 (경로 조작)


2. 대상 기능 & 엔드포인트
--------------------------------------------------------------------------------
대상 기능:
  - 게시글 작성 및 수정 (제목, 내용, 카테고리)
  - 댓글 작성 및 수정 (내용)
  - 파일 업로드 파일명

주요 엔드포인트:
  POST /api/community/posts
  PUT  /api/community/posts/{id}
  POST /api/community/posts/{id}/comments
  PUT  /api/community/posts/{id}/comments/{commentId}

접근 URL (로컬):
  http://localhost:8080/api/community/posts

테스트 계정: test@vce.com / test1234


3. 취약점 원리 (Why this works)
--------------------------------------------------------------------------------
[DoS 가능성]
서버가 대용량 요청을 처리하기 위해 메모리를 할당하고 DB에 저장 시도하는 과정에서
OOM(Out of Memory) 발생 또는 응답 지연 유발 가능.

[NULL 바이트 취약점]
  - C 언어 기반 라이브러리는 NULL 바이트(\x00)를 문자열 종료로 해석
  - Java에서도 파일명 처리 시 NullPointerException 또는 경로 절단 발생 가능
  - filename = "shell.jsp\x00.jpg" → Java String은 전체 저장, OS는 "shell.jsp"로 인식

[프론트엔드 검증 우회]
  - 브라우저에서 maxlength="100" 제한이 있어도 Burp Suite로 직접 요청 전송 시 우회
  - disabled, readonly 속성도 마찬가지로 API 직접 호출로 우회


4. 테스트 절차 (Step-by-step)
--------------------------------------------------------------------------------

[STEP 1] 제목 최대 길이 초과 테스트

Python으로 50만 자 문자열 생성 후 요청:
  python3 -c "print('A'*500000)"

Burp Repeater - 매우 긴 제목:
------------------------------------------------------------------------
POST /api/community/posts HTTP/1.1
Host: localhost:8080
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

{
  "title": "AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA[총 50만자]",
  "content": "내용",
  "categoryId": 1
}
------------------------------------------------------------------------

Burp Suite - Python 스크립트로 대용량 페이로드 생성 후 Repeater에 붙여넣기:
  import requests
  payload = {"title": "A" * 500000, "content": "test", "categoryId": 1}
  r = requests.post("http://localhost:8080/api/community/posts",
                    json=payload,
                    headers={"Authorization": "Bearer TOKEN"})
  print(r.status_code, r.headers, r.text[:200])

기대 결과 분석:
  - 400 Bad Request + "제목은 최대 N자입니다": 안전
  - 500 Internal Server Error: 취약 (예외 미처리)
  - 응답 지연(5초 이상): DoS 가능
  - 200 OK + 저장 성공: 취약 (크기 검증 없음)

5. 취약 여부 판단 기준
--------------------------------------------------------------------------------
취약 (아래 중 하나라도 해당):
  - 50만 자 제목 입력 시 200 OK로 저장 성공
  - 대용량 content 요청 처리에 5초 이상 소요 (DoS 가능)
  - NULL 바이트(\u0000) 포함 문자열이 정상 저장됨
  - 빈 문자열("") 또는 공백(" ")으로 게시글 작성 성공
  - null, {} 등 필수 필드 없이 게시글 작성 성공
  - 이모지 입력 시 500 에러 (DB 인코딩 미설정)
  - 예외 발생 시 스택 트레이스가 응답에 노출

안전 (아래 모두 해당 시):
  - 제목 최대 길이 초과 시 400 + 명확한 에러 메시지
  - 빈 문자열/공백 입력 시 400 Bad Request
  - NULL 바이트 필터링 처리
  - 이모지 정상 저장 (utf8mb4 인코딩)
  - 예외 발생 시 일반화된 에러 메시지만 반환

7. 권고 조치
--------------------------------------------------------------------------------
[Bean Validation 적용 (즉시 권고)]
// PostRequest.java
public class PostRequest {
    @NotBlank(message = "제목을 입력해주세요")
    @Size(min = 1, max = 200, message = "제목은 1-200자 사이여야 합니다")
    private String title;

    @NotBlank(message = "내용을 입력해주세요")
    @Size(max = 50000, message = "내용은 최대 50,000자까지 입력 가능합니다")
    private String content;

    @NotNull(message = "카테고리를 선택해주세요")
    @Positive(message = "올바른 카테고리를 선택해주세요")
    private Long categoryId;
}

// CommunityController.java
@PostMapping("/posts")
public ResponseEntity<?> createPost(@Valid @RequestBody PostRequest request,
                                     BindingResult bindingResult) {
    if (bindingResult.hasErrors()) {
        return ResponseEntity.badRequest().body(bindingResult.getAllErrors());
    }
    // ...
}
